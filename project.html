Problem Statement

What We Set Out to Do

We created a robot arm that is fully voice-controlled, integrating Robosuite with the Vosk speech recognition API. To benchmark the capabilities of a voice-controlled robot, we attempted two difficult manipulation tasks: nut assembly and multi-cube stacking.

To achieve fine-grained control, we designed a layered command system spanning three levels of autonomy:

AUTO

Fully Autonomous

High-level commands like "stack" and "assemble" execute the entirety of a given task without any further input.

SEG

Segmented Autonomous

Mid-level waypoint commands like "grab", "hover", and "place" let the user guide the robot through individual phases.

INCR

Incremental Control

Low-level positional commands such as "forward", "back", "up", and "down" nudge the end-effector in small discrete steps.

Speech to Text

The most important and arguably most difficult part of the project was ensuring the proper integration of a working speech-to-text model. We use the Vosk API's vosk-model-small-en-0.15 model to handle this task. At only 40 MB and requiring just 300 MB of RAM, it runs entirely on a consumer laptop alongside the physics simulation.

Raw microphone audio is read in ~0.25 s blocks. For each block we: downmix to mono → resample with polyphase resampling to 16 kHz (what Vosk expects) → peak-normalize float samples when they exceed ±1 → convert to 16-bit PCM with clipping to the valid range. Those PCM chunks are fed to Vosk's KaldiRecognizer.

Critically, we use SetGrammar to restrict the recognizer's vocabulary to only our command keywords. This dramatically improves reliability for short, single-word commands instead of open-ended transcription — the recognizer can't confuse "up" with "cup" if "cup" is not in the grammar.

Position Computation

Once a keyword is recognized, we determine where the robot should move next. The command level determines how much autonomy is given:

For incremental commands like "up", we set the target to a small offset from the current end-effector position. For high-level commands like "grab cube", the robot searches the Robosuite environment's observation dictionary for the nearest cube object, then sets the target slightly above the top of that object. The full "stack" and "assemble" commands invoke predefined multi-phase waypoint policies.

Due to the latency of voice recognition, we discovered that continuous movement was impossible to control precisely. The solution: the robot moves only in discrete steps, traveling a few units in a direction before stopping and waiting for the next command. This trades raw efficiency for far finer user control.

Control & Movement via PID

Once the robot knows its target, a PID controller drives the end-effector to that position. PID gives two key advantages over open-loop movement: it self-corrects if the robot deviates from course due to obstacles or physics errors, and it produces smooth acceleration rather than jerky steps.

Each simulation step, the controller outputs a 7-dimensional action vector: [ΔX, ΔY, ΔZ, axis-angle × 3, gripper]. The simulation runs at 20 Hz (dt = 0.05 s).

Parameter Value Rationale
Kp (Proportional) 10.0 High gain to reach target quickly within a small movement budget
Kd (Derivative) 5.0 Damps oscillation and prevents overshoot at the target position
Ki (Integral) 0.1 Small value to eliminate steady-state error without introducing drift
dt 0.05 s Matches the simulation's 20 Hz step rate

Evaluation

Results: Two Hard Tasks, Two Successes

We evaluated the robot on two canonical Robosuite tasks, designed to stress-test precision, dexterity, and spatial reasoning.

✓ Success — 30 seconds

Task 1: Multi-Cube Stacking

To make the cube stacking task truly challenging, we stacked 3 progressively smaller cubes on top of each other — testing the robot's ability to move precisely in all 3 spatial directions in a synchronized manner and grip objects of varying sizes.

The results were a resounding success. The robot completed the full stack in just 30 seconds, neatly placing each cube on the one below. As a bonus, it then stacked a very large cube on top of the entire tower — a testament to the robustness and precision of the voice-controlled system.

3 cubes stacked All 3 spatial axes Gripper precision
✓ Success — 51 seconds

Task 2: Nut Assembly

The nut assembly task is considerably harder. The robot must: align precisely with a thin nut just centimeters above the table → grip it → rotate → align again with the peg → and drop it accurately onto the peg. Even a small misalignment causes the nut to miss entirely.

We succeeded in this task as well, completing it in 51 seconds. Crucially, this task also required rotation control — using all axes of motion including yaw alignment with a separate rotation PID — showcasing the full capability of our 4-DoF voice-controlled system.

Discussion

Challenges & Lessons Learned

Building a real-time voice-controlled robot surfaced several non-obvious engineering problems. Here's what we ran into and how we addressed each one.

Warning: Speech Recognition Latency (~2 second delay)

Our initial plan was to have the robot move continuously, only changing direction when the user issued a new voice command. This seemed like the most natural control paradigm. In practice, the model would only receive the command and change course nearly 2 seconds after it was issued — a dealbreaker in a small, precise environment.

We tried two approaches to reduce this: switching from a hosted API to running the model locally, and trying different model sizes. Neither brought latency low enough for continuous control. We believe that with a much stronger GPU for inference, latency could be reduced drastically enough to enable continuous motion. For now, the system remains discrete.

Warning: Getting Speech-to-Text to Work Reliably

General-purpose ASR models failed frequently on short single-word commands like "up" or "back", confusing them with phonetically similar words or background noise. The entire audio preprocessing pipeline — sample rate, buffer size, normalization, PCM conversion — required careful tuning before Vosk could produce reliable results.

The key breakthrough was SetGrammar: restricting Vosk's vocabulary to only our command words. This transformed recognition from unreliable to robust for our use case.

Warning: Running Models Locally

Running a speech recognition model on a consumer laptop while simultaneously running a MuJoCo physics simulation was more resource-constrained than expected. Cloud ASR services offered better accuracy but added unacceptable round-trip latency. Vosk's vosk-model-small-en-0.15 (40 MB, 300 MB RAM) was the sweet spot: capable enough with grammar constraints, lightweight enough not to starve the simulation.

Solution: Discrete Step Movement

The key design insight that made everything work: each voice command moves the robot only a small, fixed increment instead of initiating continuous motion. Saying "up" moves the end-effector ~26 simulation steps in the +Z direction, then the robot stops and waits for the next command.

This turns a latency problem into a feature. Overshoot is bounded by the step size. There's no need for a separate "stop" command. The feedback loop — speak, observe, correct — is actually well-suited to the natural cadence of speech. We traded raw efficiency for dramatically improved controllability.

Future Directions

Vision-Language Models

The most promising next step is inserting a vision-language model (VLM) between the speech-to-text output and the position planning stage. This would enable complex, natural language instructions instead of single keywords — commands like "grab the blue cube and put it on the red one" would become possible without hardcoded policies.

Full 6-DoF Control

Currently the robot has 4 of 6 degrees of freedom — it lacks roll and pitch control. Adding these two axes would unlock a fully dexterous robot capable of far harder tasks, such as opening a door, turning a key, or manipulating objects at arbitrary orientations.
